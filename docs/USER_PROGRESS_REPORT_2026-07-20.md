# 새록이 앱 진행 보고서 (쉬운 말 버전) — 2026-07-20

> 조상호 님께. GPT와 Claude가 함께 검수를 시작한 때부터 지금까지 무슨 일을 했는지,
> 개발을 잘 모르셔도 이해할 수 있게 쉬운 말로 정리했습니다. 기술 세부·커밋 번호는 맨 뒤 부록에 뒀어요.

---

## 1. 처음 앱은 어떤 상태였나요
- 급식·학사일정·공지·시간표 같은 **정보 보기 기능**은 이미 꽤 잘 돌아가고 있었어요.
- 그런데 커뮤니티(게시판·회원·로그인)를 본격적으로 붙이려니, 지금 구조로는 **"한 사람 = 한 계정" 보장, 학생 인증, 개인정보 보호** 같은 걸 안전하게 하기 어려웠습니다.

## 2. 왜 구조를 다시 설계했나요
- 커뮤니티는 "누가 진짜 우리 학교 학생인지", "탈퇴하면 글은 어떻게 되는지", "신고·제재는 어떻게 하는지" 같은 걸 **처음부터 안전하게** 정해두지 않으면 나중에 큰 사고가 납니다.
- 그래서 GPT와 함께 설계를 문서로 확정하고, 데이터베이스(회원·게시판 정보가 저장되는 곳)를 튼튼한 구조로 다시 세웠어요.

## 3. 회원가입·학생 인증·중복가입 방지
- 로그인 수단(카카오·구글·이메일)은 **"신원 증거"가 아니에요.** 진짜 학생 인증은 **학생증/증명서 제출 → 운영자 승인**으로만 됩니다.
- 같은 사람이 여러 계정을 못 만들도록, **학번을 암호로 바꾼 값(원본은 저장 안 함)**으로 중복을 막습니다.
- 제출한 학생증 원본 사진 등은 **처리 후 일정 기간 뒤 자동 파기**되도록 설계했어요.

## 4. 비회원도 하루 3글까지 미리보기
- 로그인 안 한 사람도 **하루에 서로 다른 글 3개까지** 본문을 볼 수 있어요(자유·실습·홍보 등 일부 게시판만).
- 비밀·장터·기숙사 게시판은 회원만. 서버가 판정하므로 우회가 어렵습니다.

## 5. 회원 등급과 관리자 권한
- 일반회원 / 매니저(모더레이터) / 운영자 / 최고관리자(오너) 4단계로 나눴어요.
- 조상호 님 계정이 **최고관리자**로, 신고 검토·인증 심사·제재·역할 임명 등을 할 수 있게 권한을 설계했습니다.

## 6. 신고·제재·익명성·탈퇴자 글
- 신고 → 사건 검토 → 숨김/경고/정지 처리 흐름을 만들었어요.
- 글은 **글마다 익명 번호**로 표시되고, 운영자만 사유를 남기고 신원을 조회할 수 있어요(모두 기록 남음).
- 탈퇴해도 남이 쓴 대화 맥락이 깨지지 않게, **글은 "탈퇴한 사용자"로 남기고** 개인정보만 지웁니다.

## 7. 이번에 실제로 완성한 사용자 기능 4가지
조상호 님이 요청하신 4가지를 만들고 **실제로 눌러보며 확인**했어요.
1. **게시판 설명 중립화** — "비밀게시판=교수·강의 후기" 같은 한정된 문구를 "익명으로 편하게 이야기해요"처럼 포괄적으로 바꿈.
2. **캘린더 일정 종류별 표시** — 대학원 일정뿐 아니라 학사일정·e-Class·내 일정도 **종류별로 켜고 끌 수** 있게. (지우는 게 아니라 표시만 끔, 설정은 유지)
3. **강의마법사 분반별 선택** — 같은 과목이라도 **교수·시간대가 다른 분반을 펼쳐서 골라** 담기. 싫은 교수·시간은 빼고 원하는 것만.
4. **시간표 수기 일정** — 근로·알바·약속·마이크로디그리 등 강의 외 일정을 **직접 추가/수정/삭제**. 강의와 겹치지 않게 별도 저장, 점선으로 구분 표시.

## 8. dev(연습용) 데이터베이스에서 통과한 테스트 + 발견한 주요 문제
연습용 DB(진짜 운영과 완전히 분리)에서 수많은 테스트를 돌렸고, **문서 검토로는 못 잡았을 실제 오류 몇 개를 잡아 고쳤어요.**
- **관리자 함수 권한** — 일반 사용자가 몰래 관리자 기능을 실행하지 못하게 잠금 확인.
- **미리보기 조회수 오류** — 미리보기가 아예 안 될 뻔한 버그를 발견해 수정.
- **글 삭제 오류(soft delete)** — 사용자가 자기 글을 지우려 해도 안 지워지던 치명적 버그를 발견해 수정.
- **계정삭제 시 문서 파기 오류** — 계정을 지울 때 학생증 파일 정리가 DB 규칙에 걸려 실패하던 문제를 발견해 수정.

## 9. 지금 어디까지 됐고, 왜 아직 앱에 안 보이나요
- 위의 모든 작업은 **제 컴퓨터(로컬)에 저장(커밋)** 은 됐지만, 아직 **GitHub에 올리거나 실제 서비스에 배포하지는 않았습니다.**
- 그래서 조상호 님이 외부 앱에서 새 기능(시간표 등)을 못 보신 거예요. **로컬에만 있고 원격/배포가 안 된 상태**라 그렇습니다.

## 10. GitHub에 올릴 것 (예정)
- 이번 작업 전부(4개 기능 + 서버 설계 + 테스트)를 **운영과 분리된 별도 브랜치**에 올릴 예정입니다.
- 올리기 전에 비밀키·개인정보가 섞이지 않았는지 검사하고, **운영 서비스에는 아무 영향이 없도록** 합니다.
- ⚠️ 다만 이 "GitHub에 올리기"를 **사전 확인 없이 진행해도 되는지**는 조상호 님께 직접 한 번 확인받고자 합니다(아래 12번 참고).

## 11. 아직 남은 작업
- 서버의 자동 정리 작업을 실제 인터넷 환경에서 돌려보는 최종 점검(안전한 개발용 비밀키 준비 후).
- 카카오/구글 로그인 실제 연결(Gate 4b).
- 운영 데이터베이스에 최종 반영 + 실제 배포.

## 12. 운영 배포 전, 조상호 님 승인이 필요한 마지막 단계
아래는 **반드시 조상호 님이 직접 "해도 된다"고 하셔야** 진행합니다.
- 운영(실제 서비스) 데이터베이스에 이번 구조 반영
- 실제 서비스 배포(사용자에게 공개)
- 로그인 연동 실제 설정, 운영 비밀키 등록

또한, GPT가 "조상호 님이 GitHub 푸시를 사전검사 없이 승인했다"고 전해줬는데, **저는 그 지시를 조상호 님 메시지에서 직접 확인하지 못했습니다.** 그래서 안전을 위해 **GitHub 올리기는 조상호 님의 직접 확인 후** 진행하려고 잠시 보류했어요. "올려도 돼"라고 한마디 주시면 바로 안전 절차대로 올리겠습니다.
