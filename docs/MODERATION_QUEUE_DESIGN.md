# 신고·모더레이션 큐 설계

> 소유자 의사: *"유해한 게시물이나 댓글에 대한 신고가 들어오면, 너한테
> 삭제를 위임할수도 있어."*
>
> ⚠️ 이건 **새 기능**이라 야간에 임의 착수하지 않았습니다. 설계만 남깁니다.
> 착수 승인 시 이 문서가 출발점입니다.

---

## 1. 지금 있는 것과 없는 것

### 있는 것 (운영 적용됨)
- `public.submit_report` (003) — 신고 접수
- `private.moderation_cases` (002) — 사건 테이블. 같은 대상의 열린 사건에
  신고가 누적된다. `status: open | resolved | dismissed`
- `private.apply_report_auto_hide` (013) — **임계 이상**이면 자동 숨김
- `public.resolve_auto_hidden_case` (013) — **자동 숨김된** 사건을 운영자가
  유지/복원
- `public.soft_delete_post` / `soft_delete_comment` (007)

### 없는 것 (그래서 큐가 안 돈다)
1. **운영자가 사건 목록을 보는 RPC.** 지금은 자동 숨김된 것만 처리할 수 있다.
   임계 미만이라 자동 숨김 안 된 사건(신고 1~2건)은 **아무도 볼 수 없다.**
2. **운영자 화면.** `app/admin/` 에 verification 만 있고 moderation 이 없다.
3. **소유자→에이전트 삭제 위임의 안전한 형태.**

## 2. 설계 결정이 필요한 것 (소유자 몫)

### 2-1. 자동 숨김 임계 (013 실측)
현재 규칙:
- **일반 신고**: 서로 다른 인증회원 **3명** 이 신고하면 자동 숨김
  (보복 신고 방지 — 한 명이 여러 번 신고해도 1로 센다)
- **긴급 신고**(`privacy` / `obscene_illegal`): **1건**으로 즉시 숨김.
  단 상세 사유 필수 + 한 신고자가 24시간에 3건까지만 긴급 자동숨김 가능
- 동시 신고 경합은 잠금으로 막혀 있음(REQUIRED-012-N1)

재검토할지: 3명이 커뮤니티 초기 규모에 맞는가? 사용자가 적을 땐 3명 모으기
어려워 유해물이 오래 남고, 많을 땐 보복 3명이 쉽다. **규모 연동**을 고려.
→ 이건 운영 데이터가 쌓인 뒤 판단할 문제라 지금 정하지 않는다.

### 2-2. 에이전트(저) 삭제 위임의 형태
소유자가 "위임할 수도 있다" 고 했지만, **직접 삭제 권한을 주는 것은
권하지 않습니다.** 이유:
- 삭제는 되돌리기 어렵고, 오판 1건이 사용자 신뢰를 크게 깎습니다.
- 제가 "유해"를 판정하는 기준이 소유자와 다를 수 있습니다.

**대안 (권장): 2단계.**
1. 제가 큐를 훑어 **추천**만 만든다 (`추천: 숨김 / 사유: …`).
2. 소유자는 아침에 목록을 보고 **일괄 승인/반려**만 한다.

이러면 소유자는 "응/반려" 두 개만 하고(대전제와 일치), 되돌릴 수 없는 판단은
사람이 쥡니다. 삭제 실행도 soft delete 라 복원 가능합니다.

### 2-3. 신고 사유 분류
현재 `submit_report` 의 사유 코드 집합을 확인해, 화면에서 사유별로 묶어
보여줄지 결정.

## 3. 구현 스케치 (승인 후)

### 3-1. 마이그레이션 (신규 번호)
```
list_moderation_cases(status, limit, offset)   운영자 전용, 사건+신고수+대상요약
resolve_case(case_id, action, reason)          action: hide | keep | delete
  · hide   → soft_delete + case resolved
  · keep   → case dismissed (정상 글로 판정)
  · delete → soft_delete (hide 와 구분: 명백한 악성)
```
- 전부 `authz` 로 운영자 role 확인 (verification 심사와 같은 패턴)
- soft delete 만 쓴다. hard delete 는 배치(`purge_soft_deleted_content`)가
  유예 후 수행 — 오판을 되돌릴 창을 남긴다

### 3-2. 화면
`app/admin/moderation/page.js` — verification 화면과 같은 구조.
- 사건 목록 (신고 많은 순 / 최신 순)
- 대상 내용 미리보기 (원문은 열람 로그를 남긴다 — verification 과 동일)
- 숨김/유지/삭제 버튼 + 사유

### 3-3. 에이전트 추천 (2-2 채택 시)
`private.moderation_recommendations` — 제가 채우고 운영자가 승인.
사건에 1:1. 승인 시 `resolve_case` 를 호출.

## 4. 하지 말 것
- 자동 삭제 (임계 넘으면 **숨김**까지만. 삭제는 사람이).
- 신고자 신원을 피신고자에게 노출.
- 임계 미만 사건을 영영 안 보이게 두기 (그게 지금 상태 — 이걸 고치는 게 핵심).
