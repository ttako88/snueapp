# 화폐 분리 설계 (SR / 유료 / 마켓 결제)

> GPT BLOCKER 대응. `private.ticket_ledger` 하나로 모든 가치 이동을 다루면
> 성격이 다른 셋이 섞인다. 섞인 뒤에는 되돌릴 수 없다 — 원장은 append-only 이고
> 이미 기록된 행을 소급해서 분류할 방법이 없다.
>
> **작성 시점에 유료 기능은 하나도 없다.** 그래서 지금이 나누기에 가장 싼 때다.

---

## 1. 셋은 왜 다른가

| | **SR (무료)** | **유료 크레딧** | **마켓 결제** |
|---|---|---|---|
| 어떻게 생기나 | 기여·광고 시청 | 현금 구매 | 사용자끼리 |
| 소멸 | 만료 있음 | 만료 없음(원칙) | 해당 없음 |
| 환불 | 불가 | **가능해야 함** | 당사자 간 |
| 양도 | **불가** | 불가 | 해당 없음 |
| 회계 성격 | 마케팅 비용 | **부채(선수금)** | 플랫폼 무관 |
| 플랫폼이 돈을 보유 | 아니오 | 예 | **아니오여야 함** |

핵심은 마지막 두 줄이다. 유료 크레딧을 받는 순간 그 돈은 **매출이 아니라 부채**다
(사용자가 아직 안 쓴 것). 환불 요구가 들어오면 돌려줘야 한다.
SR 과 같은 잔액에 섞이면 **얼마를 돌려줘야 하는지 계산할 수 없다.**

---

## 2. 지금 당장의 제약 — 사업자등록이 없다

소유자 확인: **개인 사업자등록 없음.**

여기서 따라오는 것 (⚠️ 아래는 제 판단이고 **법률 자문이 아니다.**
실제 시행 전 반드시 확인이 필요하다):

- **유료 크레딧 판매는 지금 할 수 없다.** 계속·반복적으로 대가를 받으면
  사업자등록 대상이고, 통신판매업 신고도 별도로 필요할 수 있다.
- **에스크로(플랫폼이 돈을 잠시 보유)는 더 무겁다.** 남의 돈을 받아 보관했다가
  넘겨주는 구조는 결제대행/선불수단에 해당할 소지가 있고, 그쪽은 등록·자본금
  요건이 붙는다. **개인이 취미로 열 수 있는 문이 아니다.**
- 따라서 **식권 마켓 v1 은 "매칭만" 이어야 한다.** 플랫폼은 누가 팔고 누가
  사는지만 이어 주고, 돈은 당사자끼리 직접 주고받는다. 앱은 금액을 기록하지도
  중개하지도 않는다.

> 이건 기능을 줄이는 결정이 아니라 **문을 잘못 열지 않는 결정**이다.
> 돈을 한 번이라도 보유하면 되돌리기 어려운 의무가 생긴다.

---

## 3. 광고 리워드 제약

구글 리워드 광고 정책상 리워드는 **양도 불가 · 플랫폼 내부용**이어야 한다
(소유자도 이 방향으로 확정). 그래서 SR 은:

- 사용자끼리 주고받을 수 없다
- 현금·현물로 바꿀 수 없다
- 앱 안에서만 쓴다 (강의평 열람, 지도안 생성 등)

**SR 을 유료로 팔면 이 전제가 깨진다.** "광고 보면 주는 것" 과 "돈 주고 사는 것"
이 같은 화폐면, 리워드가 사실상 현금 가치를 갖게 된다. 그래서 §1 의 분리는
회계 문제이기 전에 **정책 위반을 피하는 문제**다.

## 4. 설계

### 4-1. 원장을 화폐별로 나눈다

`private.ticket_ledger` 에 `currency` 를 넣고 **기본값을 `free` 로** 박는다.
기존 행은 전부 무료 기여 보상이므로 소급이 정확하다.

```
currency: 'free' | 'paid'
```

`marketplace` 는 **값에 넣지 않는다.** 플랫폼이 돈을 보유하지 않기로 했으므로
마켓 거래는 원장에 들어올 일이 자체가 없다. 값을 미리 만들어 두면 나중에
누군가 "이미 있으니까" 하고 쓰게 된다.

### 4-2. 잔액을 화폐별로 따로 센다

`my_ticket_balance()` 가 지금은 `sum(delta)` 하나를 준다. 이걸 나눈다.

```json
{ "free": { "balance": 120, "spendable": 120 },
  "paid": { "balance": 0, "spendable": 0 } }
```

### 4-3. 소비 순서 — 무료부터

같은 기능에 둘 다 쓸 수 있다면 **무료를 먼저 태운다.**
- 사용자에게 유리하다 (유료는 환불 가능한데 무료는 아니므로)
- 환불 계산이 단순해진다 (남은 유료 = 산 것 − 쓴 것)

### 4-4. 섞이지 못하게 DB 가 막는다

- 한 소비 건은 한 화폐만 쓴다. 두 화폐를 합쳐 하나의 값을 치르지 않는다
  (그러면 부분 환불 계산이 불가능해진다).
- 무료 잔액이 모자라면 **유료로 자동 대체하지 않는다.** 사용자가 명시적으로
  고르게 한다. 돈이 나가는 선택을 대신 하지 않는다.

## 5. 단계

| 단계 | 내용 | 지금 가능? |
|---|---|---|
| 1 | 원장에 `currency` 추가 · 잔액 분리 (022) | ✔ 가능 |
| 2 | SR 획득·소비 경로를 `free` 로 명시 | ✔ 가능 |
| 3 | 식권 마켓 = **매칭 전용** (금액·결제 없음) | ✔ 가능 |
| 4 | 유료 크레딧 판매 | ✘ 사업자등록 후 |
| 5 | 에스크로 | ✘ 별도 검토 (권장하지 않음) |

**오늘 밤 범위는 1단계다.** 나머지는 문을 열지 않은 채 자리만 만들어 둔다.

---

## 6. 아직 정하지 못한 것

- **SR 만료 기간.** 만료가 없으면 부채처럼 쌓이고, 짧으면 사용자가 화낸다.
  기여 보상은 만료 없이 두고 **광고 리워드만 만료**를 두는 안이 있는데,
  둘을 구분하려면 원장에 획득 경로가 남아야 한다 (`reason` 으로 구분 가능).
- **유료 크레딧 단가.** 사업자등록 전이라 미정.
- 마켓 매칭에서 **분쟁이 났을 때** 플랫폼이 무엇을 하는가. 돈을 안 만지므로
  법적 책임은 낮지만, 신고·차단은 있어야 한다.
