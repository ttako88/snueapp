## 12.1 한 페이지 결론

**현재 성숙도.** 이 앱은 두 개의 서로 다른 성숙도를 가진 앱이 한 몸에 있다.
정보 도구 축(급식·학사일정·공지·시간표·학점계산기·e-Class)은 **기능적 완성도가 높은 베타 수준이며, 운영 기반(자동화 테스트·CI·모니터링·환경 분리·약관/개인정보처리방침·백업 검증)을 보완하면 출시 가능한 상태에 가깝다** — 외부 데이터는 서버 프록시(`app/api/*`)로 경계가 잘 잡혀 있고, 개인 데이터는 전부 기기(localStorage)에만 있어 서버 장애·개인정보 리스크가 구조적으로 낮다. "기능 완성"과 "운영 준비 완료"는 별개이며, 후자는 Gate 2·7에서 채운다.
커뮤니티 축(로그인·게시판)은 **동작하는 프로토타입**이다 — 이메일 로그인, 익명 글, 소유권 분리(`post_owners`), soft delete까지 기본기는 갖췄지만, 회원 신원·계정 상태·역할·신고·감사라는 "운영"의 뼈대가 전혀 없다.

**지금 리팩터링이 필요한 이유.** 데이터가 사실상 0(운영자 본인 계정 1개, soft-delete된 테스트 글 1건)이고 실사용자가 없는 지금이, 회원 모델·인증 구조라는 되돌리기 비싼 결정을 바꿀 수 있는 **마지막 저비용 시점**이다. 카카오 로그인과 학생 인증을 현 구조 위에 그냥 얹으면 "1인 다계정"이 데이터에 고착되고, 소급 병합은 계정 탈취 위험 때문에 사실상 불가능해진다.

**전면 재작성 여부: 불필요, 반대.** 정보 도구 축은 손댈 이유가 없다. 커뮤니티 축도 테이블 5개·화면 4개 규모라 "갈아엎기"가 아니라 "기반 아래에 새 층을 깔기"가 맞다. 재사용되는 것: 화면 전부, API 프록시 전부, RLS-first 패턴, 소유권 분리 패턴, soft delete 패턴. 교체되는 것: profiles 중심의 얕은 회원 모델 → 회원·상태·역할·신원 모델.

**가장 위험한 문제 5개** (상세: 12.5)

1. **비회원 미리보기를 현행 패턴으로 구현하면 본문 전체가 뚫린다** — 지금 구조는 "브라우저가 Supabase에 직접 질의 + RLS가 차단"인데, 미리보기를 위해 anon에게 select를 열어주는 순간 "하루 3글" 제한은 화면 장식이 되고 API 직접 호출로 전 게시판을 긁을 수 있다. 서버 판정 경계가 필수다.
2. **회원≠로그인수단 분리가 없는 상태에서 카카오를 붙이면 1인 다계정이 고착된다** — 지금은 auth.users=회원이다. 제공자별로 계정이 생기면 제재·중복방지가 무력화되고 나중에 못 합친다.
3. **인증자료(학생증) 취급이 이 앱 최대의 개인정보 사고면이 된다** — 아직 미구현이라 지금은 위험이 0이지만, 공개 버킷·긴 보존·넓은 접근으로 만들면 단 한 번의 실수가 실명+학번+사진 유출이 된다. 설계 단계에서 경계를 확정해야 한다.
4. **감사로그 없이 운영을 시작하면 소급이 불가능하다** — 신고 처리·신원 조회·제재의 기록 구조가 없다. 나중에 붙이면 "그 전에 일어난 일"은 영원히 공백이다.
5. **개발과 운영이 같은 Supabase 프로젝트다** — 지금까지의 모든 SQL 실험이 운영 DB에서 이뤄졌다. 실사용자가 생기면 같은 습관이 곧 장애·유출이다. (`schema.sql` 재실행 사고는 경고 헤더로 1차 완화됨)

**가장 먼저 해야 할 일 5개**

1. 이 보고서 검토·승인 (Gate 1) — 특히 12.13의 결정 질문들
2. 개발용 Supabase 프로젝트 분리 + `.env` 체계 정리 (Gate 2)
3. 최소 회귀검증(스모크 테스트)을 저장소 안으로 — 지금 검증 스크립트는 전부 세션 임시폴더에 있어 재현 불가 (Gate 2)
4. 회원·신원·상태 기반 마이그레이션 확정 + 승인된 1회 초기화 (Gate 4a)
5. 카카오 로그인 + Supabase identity linking 통합 (Gate 4b)

### 데이터 흐름 (두 개의 서로 다른 경계 패턴)

**패턴 A — 정보 축 (서버 프록시형):**
`브라우저 → app/api/* (Next 서버) → 학교 공개 엔드포인트 → 파싱된 JSON 반환`
학교 서버 주소·파싱 로직이 서버에만 있고, 브라우저는 정돈된 JSON만 받는다. e-Class는 SSRF 방지(허용 접두사 검사)까지 있다. **이 패턴이 이 저장소가 이미 증명한 "서버 경계"의 모범**이고, 비회원 미리보기(12.9)가 그대로 재사용해야 할 형태다.

**패턴 B — 커뮤니티 축 (클라이언트 직접 + RLS형):**
`브라우저 → supabase-js → PostgREST → RLS가 행 단위 차단`
Supabase 호출은 정확히 5개 파일, 17회에 국한된다: `login/page.js`(2) `board/[slug]/page.js`(2) `board/[slug]/new/page.js`(1) `board/[slug]/[id]/page.js`(8) `lib/useAuth.js`(4). 인증된 회원의 CRUD에는 이 패턴이 유효하다(경계가 진짜로 DB에 있음). 문제는 이 패턴이 **"요청자가 로그인한 본인"일 때만 성립**한다는 것 — 비회원 판정·운영자 권한·신원 조회는 이 패턴으로 표현할 수 없다 (12.6).

### 현재 게시판·익명성·소유권 흐름 (잘 만들어진 부분)

- 글/댓글에 작성자 id가 **저장되지 않는다**. 표시용 닉네임만 서버 트리거가 강제 기입(`handle_new_post` — 클라이언트 위조 불가), 익명이면 null.
- 작성자 연결은 `post_owners`/`comment_owners` 별도 테이블 + RLS "본인만 select". 화면의 수정/삭제 버튼 노출 판단(`isOwner`)도 이 RLS 특성을 그대로 이용한다(`board/[slug]/[id]/page.js`) — 서버 로직 없이 안전한 영리한 패턴.
- 하드 delete는 정책 자체가 없어 아무도 못 한다. 삭제는 `deleted_at` 갱신뿐.
- 읽기·쓰기 전부 authenticated 한정. anon은 빈 배열만 받는다(실측 확인됨).

**한계**: 이 익명 구조는 "일반 회원에게 익명"까지만 달성한다. 운영자의 신원 확인 경로는 **존재하지 않는다** — post_owners의 RLS가 본인 외 전부 차단하므로, 지금은 악성 익명글이 올라와도 운영자가 DB 콘솔에 직접 들어가 보는 것 외엔 방법이 없고 그 행위는 어디에도 기록되지 않는다 (가설 9 판정 근거).

## 12.6 목표 아키텍처 제안

### 핵심 원칙: "경계 패턴은 이미 둘 다 저장소 안에 있다 — 각각 맞는 곳에 쓰자"

이 저장소는 이미 두 경계 패턴을 성공적으로 운용 중이다 (12.2). 목표 아키텍처는 새 패턴 발명이 아니라 **배치 원칙의 명문화**다:

|---|---|---|
| 로그인 회원의 자기 데이터 CRUD (글·댓글·프로필) | **B: 클라이언트→RLS** (현행 유지) | 경계가 DB에 있어 화면 코드가 뚫려도 안전. 서버 왕복 없음. 이미 검증됨 |

- **채택 안 함: repository 패턴/ORM/전면 서비스 레이어.** 이유: 소비자가 화면 하나뿐인 쿼리를 감싸는 것은 파일만 늘린다. 규칙은 "**같은 쿼리·정책의 두 번째 소비자가 생기는 순간 lib으로**" — 예: 글 목록 쿼리는 이미 소비자가 2곳(목록·향후 검색)이 되므로 이동 대상이고, 시간표 로직은 이미 lib에 있으니 그대로.

- **채택 안 함: 모노레포/패키지 분리, 상태관리 라이브러리.** 현 규모에서 순수 비용.

## 12.7 인증·회원·학생 신원 모델

### 핵심 제안: 회원 본체를 자작하지 말고 Supabase의 계정 구조를 그대로 회원 본체로 쓴다

지시서 4.3은 "회원과 로그인 수단의 분리"를 요구한다. 이를 별도 `members`↔`identities` 매핑 테이블 자작으로 구현할 수도 있지만, **Supabase Auth가 이미 그 구조다**: `auth.users`(계정 1) ↔ `auth.identities`(제공자별 N) + `linkIdentity()` API로 로그인한 계정에 카카오/이메일 수단을 추가 연결할 수 있다.

따라서:

- **회원 본체 = `auth.users` 1행 = `members` 확장 테이블 1행 (1:1)**. members에 인증 상태·제재 상태·역할·기한을 둔다 (상태 2축 분리는 아래 상태 모델 참조).
- **로그인 수단 = `auth.identities`** (Supabase 관리). 카카오·Google·이메일이 같은 user에 연결된다.
- **동일 이메일 자동 연결에 대한 정정 (v1.1, GPT 검수 반영)**: 초판은 "자동 연결을 반드시 비활성화"라고 썼으나, 공식 문서 확인 결과 **검증된 동일 이메일의 자동 연결은 Supabase의 기본 동작이며 이를 끄는 설정은 문서에 존재하지 않는다** (docs/guides/auth/auth-identity-linking — "automatically links identities with the same email address to a single user", 미검증 이메일은 탈취 방지를 위해 자동 연결 안 됨). 따라서 설계 불변식을 다음으로 재정의한다:
  1. **자동 연결 여부와 무관하게, 이메일·로그인수단은 학생 신원의 증거가 아니다.** verified 상태는 오직 증빙 제출→operator/owner 승인→학번 HMAC 중복 검사 통과로만 부여된다.
  2. 자동 연결의 실제 동작(매직링크↔Google↔카카오 교차, 이메일 미제공/미검증 케이스, 타 계정 기연결 identity 재연결)은 **Gate 4b에서 dev 프로젝트로 전수 테스트**하고, 결과 확인 전에는 "동일 이메일이면 반드시 한 계정"이라고 단정하지 않는다.
  3. 수동 연결(`linkIdentity`)은 기본 비활성이며 대시보드에서 명시적으로 켜야 한다(문서 확인). "다른 이메일의 수단 추가"는 이 경로만 사용: 로그인 상태에서만 실행, 신규 제공자 인증 성공 필수, 타 member 기연결 여부 확인, 연결/해제 전후 audit log, 연결 해제 시 마지막 수단이 남지 않도록 차단, 민감 시점엔 재인증 요구 검토.
- 이 선택의 이점: 세션·토큰·재발급·연결해제를 전부 Supabase가 처리. 자작 매핑 대비 코드 1/10.
- 이 선택의 한계(정직하게): 두 auth.users를 **하나로 합치는 병합 기능은 없다**. 그래서 "중복 계정을 만들지 못하게 막는 것"(학번 유일성)이 더욱 전제 조건이 된다 — **중복가입 방지의 최종 방어선은 이메일이 아니라 학번 HMAC이다.** 이메일 자동 연결은 편의 기능일 뿐 보안장치로 취급하지 않는다.

### 로그인 수단 (v1.1 개정: Google 추가, 네이버 결론 수정)

**필수 우선 구현 (Gate 4b)**: 카카오 + Google + 이메일 매직링크. 어떤 수단으로 가입하든 최초 상태는 미인증(pending)이며, OAuth 프로필의 이름을 법적 실명으로 신뢰하지 않고, 이메일 도메인만으로 인증하지 않는다.

- **카카오**: Supabase 공식 provider 존재. 필요 작업 = 카카오 개발자 앱 생성(개인 가능) → REST 키/시크릿을 Supabase에 등록 → 콜백 URL 등록. 주의: 사용자가 이메일 제공 동의를 안 하면 이메일이 없을 수 있음 → 이메일을 회원 식별에 쓰지 않는 우리 모델에서 차단 사유는 아니나, 복구 수단 확보를 위해 가입 후 이메일 수단 추가 연결을 배너로 유도.
- **Google**: Supabase 공식 provider 존재(docs/guides/auth/social-login/auth-google). 카카오와 동일 원칙. Google Cloud 콘솔에서 OAuth 클라이언트 생성 필요(개인 가능).
- **이메일 매직링크**: 현행 유지. `migrations/002`의 도메인 제한 트리거는 **신원검증 도입(Gate 4a)과 함께 폐기** — 그때까지는 외부인 차단 울타리로 유지(폐기 시점을 Gate 4a 완료 기준에 명시).
- **네이버 (초판 결론 정정)**: 초판은 "OIDC 미지원이라 사실상 불가"라 단정했으나, 공식 문서 확인 결과 **Supabase는 임의의 표준 OAuth2 제공자를 `custom:naver` 형태로 등록하는 것을 지원한다** (docs/guides/auth/custom-oauth-providers — Authorization/Token/UserInfo URL 수동 입력, OIDC discovery 불필요, Free 플랜 3개까지). 네이버도 표준 OAuth2+userinfo API를 제공하므로 기술적으로 닫혀 있지 않다. **결론: "불가"가 아니라 "검증 후 결정"** — Gate 4b에서 dev 프로젝트에 `custom:naver`를 구성해 짧은 feasibility test(토큰 교환, userinfo 매핑, 고유 ID 안정성, 이메일 제공·검증 신뢰성)를 실시하고, 단순·안전하게 되면 추가, 비표준 우회가 필요하면 사유·재검토 조건을 문서화하고 연기.

### 학생 신원 (핵심 테이블 3개)

```
members            (id=auth.users.id PK, verification_status, sanction,
                    sanction_until, role, verification_deadline,
                    nickname_changed_at, created_at)
                    ← profiles를 확장·개명. 닉네임은 공개 프로필 성격이라
                      여기 두되, 아래 school_identities와 절대 분리
school_identities  (member_id PK·UNIQUE → members,
                    student_no_hmac UNIQUE,   ← 학번 원문 아님 (아래)
                    real_name, verified_at, revoked_at)
                    ← RLS: 전면 차단. 서버(definer)만. 일반·모더레이터 접근 불가
verification_requests
                   (id, member_id, doc_type, storage_path, status,
                    submitted_at, reviewed_by, reviewed_at, reject_reason,
                    purge_after)
                    ← 자료 메타. 원본은 비공개 Storage 버킷
```

- **학번 유일성 vs 비노출**: `student_no_hmac = HMAC-SHA256(서버 전용 비밀키, 정규화된 학번)`. DB가 통째로 유출돼도 키 없이는 학번 역산 불가, 그러나 중복가입 판정(UNIQUE)은 가능. "암호화한 척"이 아닌 실효 조치이며 비용은 서버 비밀키 1개 관리뿐. 학번 원문은 저장하지 않는다. HMAC 계산은 서버/Edge Function에서만, 비밀키는 클라이언트·저장소 비노출. **HMAC도 개인과 연결되는 가명 식별값으로 취급한다("원문이 아니니 개인정보 아님"이라 전제하지 않음).**
- **HMAC 키 교체 전략 (v1.2 신설, Gate 3 설계 항목)**: 키가 교체되면 같은 학번도 다른 HMAC이 되어 중복 방지가 깨진다. → `hmac_key_version` 컬럼 저장, 현재/이전 키의 교체 절차(신규는 새 키, 대조는 구 키 버전으로 재계산), 키 유출 시 재계산·마이그레이션 절차, 키 백업·복구 절차를 문서화. 로그에 키·학번 원문·계산 전 입력값을 남기지 않는다. 초기부터 복잡한 키 관리 시스템은 만들지 않되, key_version과 절차 문서는 001부터 포함.
- **실명 (v1.1 확정: 보관)**: 악성 사용자 대응·운영 책임성을 위해 승인 후에도 보관하되 — `school_identities`(RLS 전면 차단, definer 경유만)에 격리, 게시글·댓글·프로필 등 일반 API 응답에 절대 미포함, moderator 열람 불가, operator/owner만 사건 연결+사유 입력+audit log 경유 조회, 운영 콘솔에서 목록 일괄 실명 노출 금지, 개인정보처리방침에 목적·보존기간 명시. 실명을 닉네임·익명 표시명으로 사용하지 않는다.
- **인증자료 보존 2단 정책 (v1.1 세분화)**:
  - *최대 7일 내 삭제*: 학생증 사진·Smart ID 캡처·증명서 원본(Storage object), storage_path, 문서 속 부수 개인정보(생년월일·사진·주소 등), 장기 보관 사유 없는 상세 메모.
  - *최소 유지 기록*: 인증 수단(method)·상태·submitted_at/reviewed_at/approved_at·reviewer_id·마지막 재인증 시각·재인증 필요 시점·표준화된 반려 사유 코드·학번 HMAC. → "언제 어떤 방식으로 인증됐는가"는 남고, 민감 원본만 사라진다.
- **인증자료 Storage (v1.2 표현 정정 — "메타까지 삭제" 아님)**: 비공개 버킷, 업로드는 본인 폴더만(RLS), 열람은 operator/owner 검토 화면에서 서버가 단기 서명 URL 발급. 확장자가 아닌 실제 콘텐츠 MIME·크기 서버 검증. `purge_after`(=최종 검토 후 원본 삭제 예정 시각) 경과 시 배치가: ①Storage 원본 삭제 → ②성공 확인 후 storage_path를 null로 → ③원본 MIME·파일명·상세 메모 등 민감/불필요 메타만 제거 → ④**method·표준화 status·submitted_at·reviewed_at·reviewer_id·사유 코드 등 최소 감사기록은 정책 기간 동안 유지**. `submitted`(심사 대기) 상태의 자료는 7일 경과만으로 자동 삭제하지 않는다. 계정 삭제·요청 철회·장기 미처리·고아 업로드 각각의 파기 기준은 Gate 3에서 명시.
- **중복 학번 발견 시**: "이미 인증된 학번입니다. 본인이 맞다면 기존 로그인 수단으로 로그인하거나 운영자에게 문의하세요"만 표시 — 기존 계정의 닉네임·이메일 등 존재 정보를 노출하지 않음 (지시서 4.3). 자동 병합·자동 승인 절대 금지.
- **재가입 정책 (v1.2 보완)**: ①미인증 만료로 삭제된 계정 → 흔적 없이 재가입 자유 ②rejected(재제출 기한 내) → 새 계정이 아니라 기존 계정에서 재제출 ③일반 탈퇴(제재 없음·열린 사건 없음) → 학번 HMAC 삭제, 재가입·재인증 자유 ④**활성 제재 중 탈퇴 또는 banned** → 아래 enforcement hold로 동일 학번 재인증 차단(최소한 제재 종료 시점까지) ⑤**미해결 신고·열린 moderation_case가 있는 탈퇴**는 즉시 "일반 탈퇴"로 분류하지 않고, 사건 종결까지 hold 적용 여부를 검토 — 종결 후 제재 없으면 보존 근거 소멸로 제거.
- **⚠️ 설계 충돌의 명시와 해법 (v1.2)**: "Auth 삭제 시 FK cascade로 members·school_identities 정리"와 "banned의 HMAC 보존"은 **현 모델로 동시에 성립하지 않는다** (school_identities가 members에 종속돼 함께 지워짐). → Gate 3에서 **별도 최소 보존 테이블 `enforcement_holds`** 로 확정한다: members와 FK로 묶이지 않는 독립 테이블, 필드는 최소 가명정보만 — `student_no_hmac, hmac_key_version, hold_reason(sanction_type), retained_at, retention_until, source_case_id, released_at`. 실명·이메일·닉네임·OAuth 프로필·학생증 원본·사건 본문은 **보존하지 않는다**. `retention_until` 만료분은 자동 파기 배치로 삭제. 보존 기간은 법적 검토 전 "영구"를 기본값으로 두지 않는다(12.13 질문).

### 상태 모델 — 인증과 제재의 2축 분리 (v1.1 전면 개정)

초판은 suspended/banned를 인증 상태와 한 enum에 섞었다. GPT 검수 지적대로 "학교 구성원으로 인증되었는가"와 "지금 제재 중인가"는 별개 질문이므로 **두 컬럼으로 분리**한다. 하나의 status 컬럼에 인증·역할·제재를 섞지 않는다.

```
members.verification_status : pending → submitted → verified / rejected(→재제출→submitted)
                              + expired(재인증 필요 시 예약) + deleting(삭제 파이프라인 진입)
members.sanction            : none / write_restricted / community_suspended / banned
members.sanction_until      : 기한부 제재 만료 시각 (만료 배치가 none으로 복귀)
members.role                : member / moderator / operator / owner
```

- **경고(warning)는 sanction 상태가 아니다 (v1.2 정정)**: 경고는 권한을 제한하는 "현재 상태"가 아니라 "과거 조치의 기록"이다. sanction enum에 두면 복수 경고 이력 표현 불가·새 경고가 이전 것을 덮어씀·해제 시점 불명 문제가 생긴다. → 경고는 `moderation_actions`에 **불변 이력**으로 기록하고 `operational_messages`로 회원에게 표시(확인 시각 별도 저장 검토), 누적 횟수는 moderation_actions에서 집계한다. 경고 중인 회원의 실효 권한은 sanction=none과 동일.

- 실효 권한 = `role × verification_status × sanction`의 곱 (12.8 표 + 부록 E 상태표).
- 탈퇴는 상태가 아니라 **삭제 파이프라인 실행**으로 처리 (아래).
- 7일 규칙: `verification_deadline` 컬럼 + 일일 배치가 `deadline < now() AND verification_status IN ('pending','rejected')`만 삭제 대상으로 마킹. submitted는 조건에 없으므로 **검토 중 시계 정지가 자동 성립**. 반려 시 deadline을 반려일+7일로 재설정. 삭제 3일/1일 전 배너는 화면이 deadline을 읽어 표시.
- 모든 전이는 단일 definer 함수(`transition_member_status`)로만 — 경합·이중전이를 DB에서 차단하고 전이마다 `status_history` 기록.

### 미인증 계정 삭제 파이프라인 (v1.1 신설 — "SQL 한 줄" 금지)

초판의 "pg_cron이 행 삭제"는 불충분하다: members 행 삭제는 auth.users를 지우지 않으며, 사용자 소유 Storage 파일이 남아 있으면 Auth 삭제가 실패할 수 있다(docs/guides/auth/managing-user-data). 다음의 **재시도 가능·중복실행 안전한 서버 예약 작업**으로 설계한다 (실행 주체: Vercel Cron 또는 Supabase Edge Function 스케줄, service_role 키는 서버 환경변수에만):

1. **대상 조회**: 가입 7일 경과 + 인증 미신청(또는 반려 후 7일 경과) + `deleting` 아님 + 보존 사유 없음
2. **차단 마킹**: `verification_status='deleting'` → RLS·화면이 즉시 신규 제출·활동 차단 (중복 실행 방지 겸용)
3. **Storage 정리**: 인증 원본·임시 업로드·중단된 업로드 삭제 → 성공 확인 후 다음 단계
4. **참조 정리**: storage_path 제거, 서명 URL 캐시 무효화 검토, 불필요 메타 제거
5. **Auth 삭제**: Admin API로 auth.users 삭제 → FK cascade로 members·identities 정리 확인
6. **세션 검증 (Gate 4a 테스트 항목)**: 삭제된 사용자의 미만료 JWT로 API 접근이 가능한지 실측 — RLS 정책이 members 행 존재를 전제하도록 작성해 행이 사라지면 토큰이 살아 있어도 차단되게 하고, JWT 만료시간 설정을 함께 결정. 필요시 삭제 전 sanction='banned' 선적용.
7. **최소 운영 기록**: 실행 시각·성공/실패·실패 단계·재시도 횟수·비식별 내부 참조값만 로깅 (개인정보 미포함)

### owner 부트스트랩 (v1.2 보완 — "role 한 줄 업데이트"로 끝나지 않음)

프론트 하드코딩 금지(지시서 4.8). 실효 권한은 `role × verification_status × sanction`이므로 role만 owner면 pending 상태라 관리자 기능을 쓸 수 없다. Gate 4a 직후 **부트스트랩 절차 전체**:

1. Auth 계정 생성/로그인 (조상호님)
2. 실명·학생 신원 확인 (자기 자신에 대한 수동 확인 — 1인 운영기의 한계를 기록으로 남김)
3. `school_identities` 생성 + `student_no_hmac` 생성 (서버에서)
4. `verification_status='verified'`, `role='owner'`, `sanction='none'` 설정
5. 부트스트랩 실행 사실을 기록 — audit_logs가 Gate 6에서 완성되므로, 그 전에는 **부트스트랩 전용 기록**(별도 로그 테이블 또는 status_history)에 남기고 Gate 6에서 정식 audit log로 이관·보존
6. 개인 Auth ID·학번 원문은 저장소·마이그레이션 파일에 커밋하지 않음 (운영 DB 1회성 실행)

이후 역할 변경은 owner 전용 RPC+감사로그로만.

## 12.8 역할·권한표 (v1.1 — 원안 4역할 확정, 사용자 결정)

**역할 4개: member / moderator / operator / owner.** 초판의 3역할 축소안은 기각됨(12.12 참조).

**권한의 집행 구조 (v1.2 정정)**: 권한표는 *개념적으로* 하나이지만, JS 파일이 PostgreSQL RLS를 직접 지배할 수는 없다. 따라서:
- **보안 집행의 최종 기준 = DB** (RLS + 제한된 security definer 함수 + 제약조건). 서버는 DB 정책을 우회하지 않으며, service_role이 필요한 경로는 명시적 서버 전용 함수·최소 범위 쿼리로 한정하고 자체 권한 검사를 심층 방어로 추가한다.
- **`lib/policy/permissions.js` = 같은 권한표를 반영하는 UX용 미러** (버튼 표시·메뉴·안내 문구). 보안 경계가 아니며, 프런트 판정 통과가 실제 허용을 의미하지 않는다.
- **드리프트 방지**: Gate 3에서 역할×상태×제재 권한 행렬을 서면 확정 → Gate 4a 이후 DB 정책 테스트로 실제 허용·거부를 검증 → permissions.js의 예상 결과와 DB 실제 결과를 자동화 테스트로 대조, **불일치 시 DB가 최종 기준**. (한 소스에서 SQL·JS를 자동 생성하는 방식은 현 규모에 과도한 추상화라 도입하지 않음)

역할별 기본 책임: **member**=일반 이용·본인 콘텐츠·신고·차단·본인 계정 관리 / **moderator**=신고 1차 분류·숨김/복구·경고·명백한 스팸의 단기 조치 (신원·실명 열람 불가) / **operator**=인증 심사·중대 신고·기한부 제재·사유 있는 신원 조회(감사 필수) / **owner**=역할 임명·영구정지·시스템 설정·권한 정책·감사 감독. **owner가 일상 심사·신고 처리를 전담하는 구조를 만들지 않는다** — 단, 운영 인력이 1인인 초기에는 조상호님 계정이 owner이면서 operator의 권한을 함께 보유한다(역할 우회가 아니라 권한표상 owner ⊇ operator로 정의 — 인력이 생기면 위임).

⭕ 가능 / ❌ 불가 / 🔸 조건부. (상태·제재에 따른 감산은 부록 E 상태표가 우선)

| 기능 | 비회원 | member(verified) | moderator | operator | owner |
|---|---|---|---|---|---|
| 일반 게시물 이용(열람·작성·추천) | 🔸 미리보기만(12.9) | ⭕ | ⭕ | ⭕ | ⭕ |
| 신고 접수 | ❌ | ⭕ | ⭕ | ⭕ | ⭕ |
| 콘텐츠 숨김·복구 | ❌ | ❌ | ⭕ | ⭕ | ⭕ |
| 경고 | ❌ | ❌ | ⭕ | ⭕ | ⭕ |
| 글쓰기 제한(write_restricted) | ❌ | ❌ | 🔸 명백한 스팸·도배에 1일 한정 | ⭕ | ⭕ |
| 7일/30일 이용정지 | ❌ | ❌ | ❌ | ⭕ | ⭕ |
| 영구정지(banned) | ❌ | ❌ | ❌ | ❌ | ⭕ |
| 학생 인증 심사(승인·반려) | ❌ | ❌ | ❌ | ⭕ | ⭕ |

### 핵심 흐름 4개

1. **비회원 미리보기 (R1의 해소, v1.1 보강)**: `GET /api/preview/boards/[slug]` → 서버가 service_role로 목록의 **허용 필드만**(게시판명·제목·작성시각·추천수·댓글수·조회수) 반환 — 전체 데이터를 내리고 프런트에서 가리는 방식 금지. `GET /api/preview/posts/[id]` → 서버가 ①boards 등급 확인 ②서명쿠키(HMAC, httpOnly)의 오늘(KST) 열람목록 확인 ③**전 게시판 합산** 서로 다른 글 3개 미만이면 본문 반환하며 열람 기록+조회수 원자 증가 (**댓글 공개 범위는 미확정 — 12.13 질문 5**. 어떤 선택이든 댓글 수는 열람 전에도 표시, 삭제·숨김·제재 콘텐츠 미반환, 반환 범위는 서버가 제한) ④같은 글 재진입·새로고침은 열람목록에 이미 있으므로 추가 차감 없음 ⑤4글째부터는 가입 안내만 반환(본문 미반환, 조회수 미증가, 내부 오류정보 비노출). 목록 조회는 차감 없음. **posts에 anon RLS를 여는 일은 없다.**
   *쿠키 삭제 우회에 대한 최소 서버 방어(GPT 1-6, v1.2 수정)*: 1차 정책 제한은 서명된 httpOnly 쿠키 기준 3글. IP 기반 방어는 3글 제한의 집행 수단이 아니라 **쿠키 반복 삭제·자동 크롤링을 억제하는 abuse backstop**으로만 둔다 — 단순 hash가 아닌 **서버 비밀키 HMAC + 일일 파생키(또는 key version)**, 원 IP 미저장, HMAC 결과·카운터는 24시간 내 삭제, 신뢰 가능한 프록시 헤더에서만 IP 추출, 상한값은 코드 고정이 아닌 **환경설정**으로 두고 공유 IP(학교 Wi-Fi·기숙사·통신사 NAT) 오탐을 피할 만큼 충분히 높게 시작해 실측 후 조정(구체 수치는 Gate 3에서 확정 — 확정값 아님). 별도의 엔드포인트 일반 rate limit도 병행. IP HMAC은 개인정보성 보안 로그로 취급해 처리방침에 기재. 상한 도달 시에도 가입 안내/일반 제한 응답만 반환. → 채택 여부는 12.13 질문.
2. **신고→사건→제재 (v1.2 — 12.8 권한표와 일치화)**: 신고 insert 시 트리거가 열린 case에 병합 or 신설 + 스냅샷 저장. **moderator** = 숨김/복구·경고·명백한 스팸의 1일 write_restricted, **operator** = 7일/30일 community_suspended·인증 심사·사건 연결 신원조회, **owner** = 위 전부 + banned·역할 관리. 모든 조치는 definer 함수가 `moderation_actions`+`audit_logs`+`members.sanction`/`sanction_until`을 한 트랜잭션에 기록.
3. **익명 신원조회 (v1.2 — 보장 범위 정직화)**: `rpc admin_reveal_author(content_id, case_id, reason)` — definer 내부에서 ①호출자가 operator/owner이며 정상 상태인지 검증 ②**연결된 신고 사건(case_id) 필수** ③reason 필수 ④audit_logs insert(조회자·시각·대상·사건·사유) ⑤owners/school_identities 조회 반환. 감사기록과 조회가 같은 트랜잭션이며, 프런트·일반 관리자 API에 다른 조회 경로를 만들지 않는다. moderator는 호출 권한 자체가 없다.
   **보장의 한계 (명시)**: 위는 **앱이 제공하는 정상 경로에 한한 보장**이다. Supabase 프로젝트 소유자·DB 관리자·service_role 서버 코드는 RLS를 우회할 수 있으므로 "구조적으로 불가능"이 아니다 — service_role은 서버 전용으로 격리하고 사용 지점을 최소화하며, 대시보드 직접 조회는 긴급상황용 break-glass 절차로 제한하고 가능한 플랫폼 로그를 확인한다. **1인 운영기에는 완전한 직무 분리가 불가능하다는 한계를 정직하게 기록해 둔다.** 요약: "앱 정상 경로에서는 감사 없는 신원 조회를 허용하지 않으며, 인프라 최고권한의 우회 가능성은 별도의 접근통제·운영정책으로 완화한다."
4. **삭제·보존**: soft delete 유지. 일배치 — 신고 사건 없는 `deleted_at < now()-30일` 영구삭제, 사건 걸린 콘텐츠는 종결 전 보존(스냅샷과 별개). ⚠️ 개인정보 삭제 요구와 증거 보존의 충돌은 **법적 검토 필요 항목** — 운영정책 문서에 "신고 처리 중 콘텐츠는 처리 완료까지 보존됨"을 고지하는 것으로 1차 대응하되, 출시 전 개인정보처리방침 작성 시 재검토 (분석 범위 밖 표시).

### 집계·페이지네이션

- vote_count/댓글수는 **트리거 유지 집계**(저장) — 목록 화면이 매번 count하는 비용 제거, 정합은 트리거가 한 트랜잭션에서 보장. view_count는 서버 경로만 증가.
- 페이지네이션: `(created_at, id)` 복합 커서 방식 (offset은 삭제·삽입 시 중복/누락 발생). 인덱스: posts(board_id, id desc), comments(post_id, id), reports(case), guest_reads(cookie_id, date).

## 12.10 마이그레이션·초기화 전략

**권장: 승인된 1회 초기화 + 마이그레이션 재시작.** 현재 실데이터 = 조상호님 계정 1 + soft-delete된 테스트 글·댓글 각 1. 보존 가치가 0에 수렴하므로, 기존 스키마와의 호환 계층을 만드는 비용(구 profiles→신 members 변환 코드, 이중 트리거)이 이득을 압도한다.

- 대안(비초기화 증분 전환)도 가능은 하다: profiles를 rename+확장, 기존 글 유지. 소요는 크지 않으나 "테스트 글 1건 보존"을 위해 마이그레이션 복잡도를 올릴 이유가 없다 — **비추천**.
- **초기화 승인의 의미 (v1.1 명확화)**: 사용자·GPT 공동 승인은 "Gate 4a에서의 실행" 승인이지 즉시 실행 승인이 아니다. **Gate 4a 이전에는 기존 DB를 변경·초기화하지 않는다.**
- **Gate 4a 실행 전 필수 안전 절차 (v1.1 신설)**:
  1. dev 프로젝트에서 전체 마이그레이션 리허설 (Storage 버킷 포함)
  2. 기존 데이터·Storage의 백업/내보내기 (pg_dump)
  3. 현재 연결 대상이 dev인지 운영인지 명시적 확인 + **초기화 대상 프로젝트 ID/URL 재확인**
  4. 초기화 후 재생성될 테이블·정책·버킷 목록 제시

## 12.12 반론과 대안 (지시서 질문에 솔직 답변)

- **과한 부분은?** ① ~~역할 4개 — 3역할 축소 제안~~ → **철회·기각됨 (v1.1)**: 사용자·GPT 공동 결정으로 원안 4역할 확정. 다만 "1인 운영기의 현실"은 권한표에 owner ⊇ operator로 반영해 해소 (12.8). ② moderator의 "제재 제안→승인" 2단계 워크플로 — moderator가 실제로 생기기 전까지 최소형 유지(경고·명백한 스팸 1일 제한만 직접, 이상은 operator/owner) — GPT 역할 정의와 합치. ③ 조회수 규칙의 정밀도 — "회원: 글당 1회, 비회원: 서버 경로만" 정도로 시작해도 목적(조작 완화) 달성. 기간별 재집계는 후순위.
- **더 단순하지만 같은 안전성?** 회원↔로그인수단을 auth.users+identity linking으로 (12.7) — 자작 매핑 대비 코드·버그 면적 대폭 축소, 안전성 동일 이상. 미리보기도 RPC 남발 대신 Route Handler 2개+서명쿠키(+IP해시 백스톱)면 충분.
- **역할 수 적절한가?** 4개 확정 (결정 사항 — 12.8).
- **member와 auth.users 분리의 최적 수준? (v1.2 정오)** Supabase의 동일 이메일 identity 자동 연결과 별개로, **이미 서로 다른 auth.users로 생성된 두 계정을 앱이 임의로 사후 병합하는 기능은 제공하지 않는다.** 따라서 별도 member ID 계층을 두어 병합 가능성을 확보할 실익보다 복잡성과 계정 탈취 위험이 더 크므로, 현재는 auth.users와 members의 1:1 구조가 적절하다 — 단, FK를 전부 members.id로 걸어두면 훗날 분리가 필요해져도 members 내부만 바꾸면 된다(화면 영향 0).
- **인증자료 직접 수취보다 나은 대안?** 검토했다: ①학교 이메일 소유 확인 — 1인 다메일 문제로 단독 부적격(단, **보조 신호**로 가치 있음: 학교메일 인증자에겐 검토 우선순위 부여 가능) ②학교 공식 연동(포털 OAuth) — 존재하지 않음, 학교 협의는 장기 과제 ③공인 재학인증 API — 초등교원 양성대에 적용 가능한 상용 서비스 없음. **결론: 자료 검토가 현실적 최선**이나, 운영 부하를 정직하게 경고한다 — 개강 시즌 하루 수십 건 제출 시 owner 1인 검토가 병목이 된다. 처리 SLA(예: 48시간)를 미리 공지하고, 반려 사유 템플릿을 준비할 것.
- **카카오 도입이 일으킬 문제? (v1.2 수정)** ①이메일 미제공·미검증 계정의 복구수단 부재(→수단 추가연결 유도) ②동일 이메일 자동 연결의 실동작을 dev에서 검증해야 하며 예상 밖 identity 연결 가능성을 테스트해야 함 — **OAuth 연결 여부와 학생 인증 여부는 절대 혼합하지 않고, 중복가입 방지 장치는 어디까지나 학번 HMAC** ③콜백 URL·키 관리 대상 증가 ④(마이너) 카카오 심사 시 서비스 설명 필요.
- **네이버 제외가 합리적인가?** ~~"불가" 단정~~ → **정정 (v1.1)**: Supabase가 커스텀 OAuth2 제공자를 공식 지원함을 문서로 확인 — "제외 확정"이 아니라 "Gate 4b feasibility test 후 결정"이 맞다 (12.7).
- **비회원 3글 미리보기가 보호 못 하는 것?** 결심한 크롤러(쿠키 무시 대량 IP), 열람된 글의 외부 유포·아카이빙, 스크린샷. 즉 "허용 게시판에 쓴 글은 사실상 공개"라는 지시서의 전제가 맞고, **작성 화면의 고지가 기술 장치보다 중요**하다.
- **조회수 정밀화의 복잡성?** 이미 과하다고 판단 — 위 첫 항목처럼 축소 시작.
- **관리자 신원조회의 가장 단순한 안전 구현?** definer RPC 하나에 사유검증+감사기록+조회를 한 트랜잭션으로 (12.9 흐름 3). 별도 관리자 서버·화면 없이 Gate 6 초기엔 이 RPC 하나로 충분.
- **리팩터링과 신기능의 분리 지점? (v1.2 수정)** 동작 변경 없는 코드 이동은 Gate 2, 서면 설계 확정은 Gate 3(코드·DB 무변경), 스키마 실행은 Gate 4a — 세 단계가 섞이지 않게 유지한다. 추가로 4a/4b 분할을 제안했다 (12.11).

## 12.13 사용자에게 필요한 최종 질문 (v1.1 — 해소분 정리)

**해소된 결정 (기록)**: ①DB 초기화 승인(Gate 4a에서만 실행, 12.10 안전 절차 필수) ②역할 원안 4개 확정 ③실명 보관 확정(GPT §5-3 조건부 — 격리·감사 필수) ⑦정지 회원 열람은 단일 답 대신 제재 3단계별 차등으로 확정(write_restricted=읽기 가능 / community_suspended=회원전용 열람 불가 / banned=이용 불가) ⑥커밋은 수정본 재검수 통과 후 실행(절차 확정).

**남은 결정 질문 (v1.2 갱신)**:

1. **개발용 Supabase 프로젝트 생성 승인** — `snueapp-dev`. 필요 시점: Gate 2 착수 시. 현재 무료 한도(활성 2개) 안에서는 추가 비용이 없을 가능성이 높으나 생성 직전 콘솔에서 실제 한도를 확인한다. 무료 프로젝트는 장기 비활성 시 일시정지될 수 있음. dev·운영의 환경변수/프로젝트 ID를 명확히 분리하고, **dev에는 테스트 데이터만 사용(실제 학생증 자료 복제 금지)**.
2. **카카오·Google 개발자 앱 생성 — 원칙 승인 여부** — 실제 생성은 Gate 4b 직전에, 준비표(필요 계정·콜백 URL·dev/운영 앱 분리·동의항목·키 저장 위치·redirect allowlist·키 교체 방법) 제시 후 진행. 비밀값(client secret, service_role 등)은 채팅·보고서·스크린샷·Git·NEXT_PUBLIC 변수에 절대 남기지 않음.
3. **비회원 미리보기 IP HMAC 보조 방어 채택** — 서명쿠키(3글/일)가 기본 제한이고, IP HMAC은 자동 우회 억제용 보조장치로만(서버 비밀키 HMAC+일일 파생키, 원 IP 미저장, 24h 파기, 환경설정 상한, 처리방침 기재). *GPT 의견: 수정 조건부 채택.* 상한 수치는 Gate 3에서 확정.
4. **banned·제재 중 탈퇴자의 학번 HMAC 보존 기간** — 지금 특정 연수를 확정하지 않음. Gate 3에서 retention_until 스키마·보존 사유·hold 규칙·자동 파기를 확정하고, 구체 기간은 개인정보처리방침+법적 검토와 함께 출시 전 확정. **법적 검토 전 "영구 보존"을 기본값으로 두지 않는다.**
5. **비회원이 허용된 3글을 열었을 때 댓글도 보이는가 (v1.2 신설 — 제품 정책)** —
   - A. 본문만, 댓글은 개수만 표시 (가입 유도 최대, 노출 최소)
   - B. 본문 + 일부 댓글 (예: 최신 3개 — 분위기는 보여주되 전체 공개 회피) ← *GPT 추천. 댓글 추천 기능이 없는 초기엔 "최신 3개" 단순형으로 시작 가능*
   - C. 본문 + 전체 댓글 (접근성 최대, 노출 최대)
   어떤 선택이든: 댓글 수는 열람 전 표시, 비회원 응답에 내부 id·이메일·실명 미포함, 삭제·숨김·제재 콘텐츠 미반환, 반환 범위는 서버에서 제한.
