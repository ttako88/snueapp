### 제가 낸 사고 2 — 유료 스크립트에 `--help` 를 넣었다가 배치가 돌 뻔했습니다
샘플 생성기 사용법을 보려고 `--help` 를 붙였는데, 그 스크립트엔 도움말이 없어서
**알 수 없는 플래그를 무시하고 기본 유료 배치를 그냥 시작**했습니다. 즉시 프로세스를
죽였고 — **샘플이 한 건도 안 나온 시점**이라(진행 로그·출력 파일 모두 비어 있음)
**실비는 0원**입니다. 확인: 최근 5분 내 생성된 결과 파일 없음.

원인은 제 오타가 아니라 **도구에 방어가 없던 것**이라 봤습니다. 그래서 도구 자체를
고쳤습니다: 이제 `--help` 는 사용법만 찍고 끝나고, **모르는 플래그는 실행 전에
거부**합니다(오타 한 번에 돈이 나가지 않게). 이름만 바꾼 게 아니라 기계적으로 막았습니다.

### 동시 finalize 결함 — 실증은 못 했습니다
아래 수정은 **코드 분석으로 찾은 것이고 실제로 재현해 보지는 못했습니다.**
재현하려면 운영 DB 에 인증 신청을 만들어야 하는데 그건 운영 데이터 변경이라
밤에 하지 않기로 한 항목입니다. 낮에 승인받고 재현 테스트를 돌리는 게 맞습니다.

## 2-2. 미해결 결함 2건 처리

### ① 강의평가 통계가 항목별 분모를 무시 `LOCAL_VERIFIED` (dry-run PASS)
011 의 공개 게이트는 "서로 다른 작성자 10명" 하나뿐인데, 016 의 제출 RPC 는
**5개 항목 중 2개만** 답해도 통과시킵니다. 그래서 작성자가 10명이어도 특정
항목은 3명만 답한 상태가 될 수 있고, 그 3명이 같은 값을 골랐다면 출력이
`{"보통": 3}` 이 됩니다 — **분모 10이 아니라 분모 3인 통계**입니다.
k=10 게이트가 그 항목에는 한 번도 적용되지 않았습니다.

`early`(5~9명) 분기도 최빈값 비율의 **분모가 틀렸습니다**. 항목 응답자 수가
아니라 전체 작성자 수로 나누고 있었습니다.

→ 016 에 `course_review_stats` 교체를 넣었습니다. 항목마다 응답자 수를 따로
세고, 그 수가 10 이상일 때만 분포를 공개합니다. 분모(`n`)도 함께 내려보냅니다 —
분모를 숨기면 3명짜리 통계를 10명짜리로 오해합니다.

### ② 동시 finalize 에 선점이 없음 `CODE_WRITTEN` (dry-run PASS)
finalize 는 `status='uploading'` 을 **읽어서** 확인한 뒤, 파일을 내려받아
검증하고, 같은 `verified/<id>/document` 에 `upsert:true` 로 올립니다.
읽기와 쓰기 사이에 아무 잠금이 없습니다.

같은 요청으로 finalize 가 둘 들어오면 둘 다 검사를 통과하고 각자 staging 을
내려받습니다. staging 업로드 토큰은 아직 살아 있으므로 그 사이 파일을
바꿔치기할 수 있고, 그러면 **먼저 무결성 검사를 통과한 쪽의 바이트를 나중
것이 덮어씁니다.** 정본에 아무도 검증하지 않은 바이트가 남습니다.

→ `020_finalize_claim.sql`: 파일을 건드리기 **전에** 선점합니다.
- 새 status 값 대신 **nullable 컬럼**(`finalize_claimed_at`)을 씁니다.
  status 는 7종이 여러 CHECK·부분 유니크 인덱스에 얽혀 있어 값 하나 늘리는
  비용이 크고 되돌리기 어렵습니다.
- 선점만 하고 죽는 경우를 위해 **2분 TTL**. 지나면 자동으로 풀립니다.
- 라우트의 실패 경로 **12곳 전부**를 `fail()` 하나로 모아 반드시 해제합니다.
  하나라도 빠뜨리면 그 사용자가 2분 갇힙니다.

## 6. 제가 오늘 밤 저지른 것 (숨기지 않습니다)

세 개 있습니다. 전부 발견해서 고쳤지만, 같은 종류가 또 나올 수 있어 적어 둡니다.

1. **package.json 을 정규식으로 편집**해 후행 쉼표를 남겼고 dev 서버가 잠시
   500 을 냈습니다. → JSON 은 파싱→수정→직렬화로.
2. **이미 있는 코드를 중복 작성**했습니다(`PRACTICUM_STAGES` 가 있는데
   `PRACTICUM_TERMS` 를 새로). 제가 계속 경고하던 실수를 제가 했습니다.
   → 새 모듈을 만들기 전에 같은 이름의 폴더·파일부터 볼 것.
3. **빌드 번들을 부풀리는 코드**를 넣었습니다(동적 경로 `fs` 읽기).
   빌드 경고가 잡아 줬습니다. → 새 서버 모듈을 넣으면 `npm run build` 를
   돌려 경고까지 볼 것. 구문 검사만으로는 안 잡힙니다.

공통점은 **"돌아가는 것"과 "맞는 것"을 구분하지 않은 것**입니다.
세 번 다 `node --check` 는 통과했습니다.
