## 범위: "게시 + 앱 내부 연락"까지만

앱은 거래 당사자가 아니다. 아래는 **구조적으로 만들지 않는다** — 컬럼 자체가 없어야 한다.

| 금지 | 이유 |
|---|---|
| 앱 내 결제·에스크로 | v1 범위 밖. 자금 보관은 규제 영역 |
| 구매대금 보관·정산 | 〃 |
| 계좌번호·카드정보 저장 | 저장하는 순간 유출면이 된다 |
| 앱 크레딧으로 식권 구매 | 헌장: 포인트로 재화 구매 금지 |
| 거래 성사 수수료 | 중개 책임이 무거워진다 |
| "안전거래 보증"·"정품 보증" 표현 | 보증하지 않는 것을 보증처럼 말하면 안 된다 |

대신 반드시 표시: **앱은 거래 당사자가 아니며, 실제 양도·지급·환불은 당사자 책임**.
신고·분쟁 접수 경로 제공.

## 상태기계

```
open → reserved → completed
  ├→ cancelled
  ├→ expired               (식사일 종료·당일 자정 자동)
  ├→ hidden_by_moderation
  └→ preserved_for_case
```
"completed"는 통계일 뿐, 앱이 대금지급·식권 유효성을 보증하지 않는다.

## 안전장치

- **verified 회원만** 등록·연락
- `write_restricted`: 신규 등록·연락 불가, 기존 글 취소·분쟁신고는 가능
- `community_suspended`/`banned`: 마켓 이용 불가, 이의제기만
- 같은 회원의 동시 활성 등록 상한
- 같은 식사일·식사종류 중복등록 차단
- **판매가는 액면가 이하**로 제한 (전매 차익 방지 — 우선 검토)
- 자유서술에 **전화번호·계좌번호·오픈채팅 링크 입력 차단**
- 연락은 앱 내부 식별자로만. 양쪽 모두 상대 member_id를 직접 받지 않음
- 예약·완료 전이는 행 잠금 + 멱등성키로 중복 구매 방지
- 삭제된 회원의 일반 listing은 즉시 비공개, 열린 사건 snapshot만 분리 보존

## 신고: 기존 사건 구조 재사용 (새 신고 엔진 금지)

- `target_type`에 `meal_ticket_listing` 추가
- `reason_code` 추가: `suspected_fraud`, `invalid_ticket`, `no_show`, `duplicate_sale`
- 신고 시 거래글 최소 스냅샷 저장
- 판매자 실명 조회는 기존 case+reason+audit 게이트로만. moderator는 불가
- **신고 누적만으로 자동 정지 금지.** 중복·대량 등록은 자동 flag만, 제재는 인간 검토

## 보존 (일반 게시글보다 훨씬 짧게)

| 데이터 | 정책 |
|---|---|
| 판매글 공개 | 식사일 종료 또는 당일 자정 자동 만료 |
| 취소·만료 일반글 | 즉시 비공개, 30일 후 hard delete |
| 연락 요청 | 거래 종료·취소 후 30일 |
| 사기·분쟁 사건 snapshot | 일반 데이터와 분리해 3년 보존 후 파기 검토 |
| 열린 사건·법적 보존 | 종결 또는 legal_hold 해제까지 |
| 전화번호·계좌번호 | **구조적으로 저장 금지** |

전 거래글을 3년 보존하는 게 아니라 **신고·분쟁이 발생한 최소 스냅샷만** 제한 영역에
보존한다(개인정보 최소수집).
