## 단일 BLOCKER

기존 작업자(stale worker)"를 fencing하지 못한다.** TTL은 liveness만 담당해야 하고,
무결성은 **claim token + Storage 경로 격리**로 보장해야 한다.

반례: A선점→작업시작→지연(비동기 I/O·Storage 작업 생존)→2분 후 B가 만료 선점 인수
→ A·B 둘 다 같은 `verified/<id>/document`에 upsert:true → B만 finalize돼도 A의
늦은 쓰기가 정본을 덮음. + A의 실패 정리(timestamp NULL)가 B의 새 선점까지 해제.
maxDuration=60은 HTTP 상한이지 외부 I/O·Storage 작업이 60초 내 종료된다는 fencing 증거 아님.

## MUST (재작업 스펙)
1. `finalize_claim_token uuid` 추가, claim RPC가 **새 token 반환**.
2. release·DB finalize는 `id + member_id + status + claim_token` **일치 시에만** 성공.
3. 각 claimant는 `verified/<verification_id>/<claim_token>/document` **고유 경로에
   upsert:false**로 쓴다(경로 격리 → 남의 정본 못 덮음).
4. 최종 DB tx가 **현재 token 재확인** 후 그 claimant의 고유 경로만 정본 결합.
   패배 경로는 orphan으로 남겨 별도 정리(021 탐지 대상).
5. 모든 외부 I/O에 명시적 timeout(단 timeout은 fencing 대체 아님).
6. **예외 처리(throw/finally)에서도 token 조건부 해제.** 프로세스가 죽어 해제 못 한
   경우에만 TTL 인수 사용.
7. 배포 파이프라인이 020 함수 존재·시그니처·버전을 카탈로그에서 확인한 **뒤에만**
   새 라우트 배포.
8. 라우트도 시작 시 capability/version 불일치면 안전한 **503 중단**. 구형 비선점

9. 롤백: route rollback → 의존성 0 확인 → DB 객체 제거 순서.
