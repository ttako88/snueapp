# 데이터·모더레이션·수익화 헌장 (DATA_AND_MODERATION_CHARTER)

> 상태: **초안 v0.1** (2026-07-21). owner(조상호) 아이디어 → GPT↔Claude 3라운드 브레인스토밍 수렴 결과의 문서화.
> 지위: `ARCHITECTURE_AUDIT_PHASE1.md`·`GATE3_DESIGN.md`와 함께 **상위 규범**. 향후 `MONETIZATION_ADR`과 병렬.
> **새로 추가되는 모든 "데이터 수집·관리자 화면·광고·AI·수익화" 기능은 이 헌장 검사를 통과해야 설계·배포된다.**
> 헌장과 충돌하는 하위 기능 설계는 자동 보류. 헌장 변경 시 이유·영향·사용자 고지 필요 여부·검수 이력을 기록한다.
> ⚠️ 이 문서는 **방향 합의**이며, 실제 테이블·코드는 "그 기능을 처음 쓰는 시점"에 만든다(안 쓰는 민감 테이블·권한면을 미리 만들지 않음). 동결된 Gate 4a RC(001~009)는 이 문서 때문에 재개봉하지 않는다.

각 조는 **원칙 / 금지선 / 준수증거** 3부로 구성한다.

---

## 제1조 — 목적제한 (먼저 '왜'를 적는다)

**원칙**
- 모든 데이터 항목·이벤트·관리자 조회 기능은 **구현 전** 다음을 등록해야 한다: 데이터/이벤트 이름, 수집 목적, 필수/선택, 보존기한(retention_class), 제3자 제공·위탁 여부, 접근 가능한 역할.
- 목적 없이 "일단 모아두자"는 금지. 활동정보·행태정보·가명정보도 계정과 연결되면 개인정보(PIPA)일 수 있다는 전제로 다룬다.

**금지선**
- 등록된 목적 외 재사용(목적 외 이용) / 목적 없는 상시 수집 / "혹시 몰라서" 원문 보관.

**준수증거**
- 데이터 카탈로그(이름·목적·필수여부·retention·역할) 문서 존재. 신규 이벤트 추가 PR은 카탈로그 갱신 diff를 포함.

## 제2조 — 최소수집 · 기본 비수집

**원칙**
- 목적 달성에 필요한 최소 필드만. 가능하면 원문 대신 **집계·플래그·대분류**로.
- 저장가치 있는 예(수집 O): 시간표 설정(학과·학년·학기·선택과목 — 선호 경향 분석), 캘린더 토글 사용률·소스별 노출/클릭, e-Class 동기화 성공/실패, AI도구 이벤트(ai_tool_opened·generation_requested·completed/failed·template_type·재생성수·채택여부·토큰수·지연·추정원가·오류코드·간단 만족도), 앱 품질(화면 로딩시간·API 성공률·오류코드·앱버전·OS/브라우저 대분류).

**금지선 (기본 비수집)**
- 프롬프트 원문·생성결과 원문(→ 필요 시 "내 문서 보관" 별도 기능으로 사용자 소유), 개인 일정 제목·본문, 과제명·알림 내용 원문, 학생 이름·학교/학급/행동특성, 실습일지 원문.
- 장기보관 회피: 전체 User-Agent·원 IP·기기 지문·세밀한 화면이동 전체 기록.

**준수증거**
- analytics payload에 대한 "금지 필드 스캔"(secret scan과 유사) 0건. 원문 필드 부재 테스트.

## 제3조 — 식별자 분리 (분석 ≠ 신원 ≠ 결제 ≠ 모더레이션)

**원칙**
- 목적별 식별자를 서로 다르게 생성하고, 한 API에서 상호변환하지 않는다.
- 권장 식별자: `auth_user_id`(인증) · `member_id`(커뮤니티 권한) · `analytics_subject_id`(제품 분석) · `moderation_subject_id`(신고·제재) · `billing_customer_id`(결제) · `school_identity_id`(신원) · `public_anon_alias`(익명 표시).
- 연결표는 private schema에 두고 목적별 제한 RPC로만 접근. 실명 확인 RPC만 사건을 통해 연결.

**금지선**
- `analytics_subject_id`를 관리자 화면에서 `member_id`로 검색 / billing 화면에서 게시글 열람 / moderation 화면에서 결제내역 열람 / identity 화면에서 기능 사용이력 열람.
- 범용 `get_everything_about_member()` 류 RPC / 여러 종류 내부 ID가 함께 든 CSV 다운로드 / service_role 우회.

**준수증거**
- cross-schema grant 테스트. 허용되지 않은 ID 변환 RPC 0개. API projection에 두 종류 이상 내부 ID 동시 반환 0. 연결행위 audit.

## 제4조 — 사건 중심 모더레이션 · 단계적 신원공개

**원칙**
- 모더레이션 가시성은 역할만으로 열리지 않고 **"역할 + 사건(신고) 또는 승인된 위험 트리거 + 사유"** 가 있어야 열린다.
- **Tier 0** (집계 대시보드): 개인 식별 없음. 오늘 글·댓글 수, 신고 수, 게시판별 활성도, 처리 대기 사건 수, 평균 처리시간, 제재 건수, 신고 인용·기각 비율, 반복 신고 대상 계정 수. → owner 상시 열람.
- **Tier 1** (모더레이션 대상 카드): 신고 사건이 열렸거나 승인된 위험조건을 충족한 회원만. 운영용 가명 ID(예: M-7F3A) + 활동/신고 지표(글·댓글 수, 최근 30일 활동, 받은 신고 수, 서로 다른 인증회원 신고자 수, 심사중/인용/기각 신고 수, 경고·제재 이력, 이의제기 결과, 삭제 콘텐츠 비율). **닉네임·이메일·실명 기본 비표시.**
- **Tier 2** (신고 콘텐츠 간 동일계정 연결): 신고된 글/열린 사건 안에서만 "이 신고글과 이전 신고글이 같은 계정" 판단(재범). 내부는 member_id 해석하되 화면에는 UUID·신원 미반환.
- **Tier 3** (닉네임 확인): 사건 + 사유 + operator+ + audit + 단건. 일괄 금지.
- **Tier 4** (실명 확인): 기존 `admin_reveal_author` 유지 — 사건 + 대상 콘텐츠 일치 + 구체적 사유 + operator+ + 감사기록 + 단건. 폭력·협박·개인정보 유포·사기·법적 대응처럼 계정 수준 제재로 해결 안 될 때만.

**금지선 (절대금지)**
- 전체 회원 목록 / 전체 moderation_subject 목록 / arbitrary member_id 조회 / 닉네임·학과·학년으로 위험회원 검색 / 정상 익명게시물을 회원별로 펼치기 / 일괄 신원공개 / owner용 무감사 우회 RPC / 실명·활동지표 CSV.
- **member_activity_counters·Tier1 카드는 "사건 또는 승인된 위험조건"이 있을 때만 조회 가능** — 이 한 줄이 '운영에 필요한 가시성'과 '관리자의 상시 감시'를 가르는 최상위 불변조건이다.

**Tier 1 조회 트리거** (이 중 하나 필요): 열린 신고 사건 / 활성 이의제기 / 현재 제재 검토 / 중대한 사유 코드 / 승인된 스팸·도배 탐지 규칙 / 반복 위반 조건. **접근 종료**: 사건 종결 후 30일, 이후 사건 재개 또는 새 사유 필요.

**준수증거**
- case_id 없는 Tier 1 RPC 실패 / 임의 UUID 입력 실패 / 종결+30일 경과 카드 실패 / 조회 audit / 전체목록·검색 RPC 0개.

## 제5조 — 보존 · 파기 (retention_class + purge_after)

**원칙**
- 모든 데이터는 생성 시 `retention_class`와 `purge_after`를 갖는다. `null=영구`는 원칙적으로 금지(사유 명시 예외만).
- 데이터 클래스별로 보존 시계가 다르다(분석은 짧게, 모더레이션·법적 기록은 길게).

**보존기한 초안 (법률검토 전 제품 기본값)**
| 데이터 클래스 | 보존 |
|---|---|
| 원시 제품 이벤트(raw event) | 30일 |
| 가명 분석 이벤트 | 90일 |
| 게시판별 집계 지표 | 1년 내외 |
| 오류 로그 | 30일 |
| 종결된 일반 모더레이션 사건 | 1년(초안) |
| 안전·법적 관련 사건·제재 | 사유 존속 기간(더 길게, 법률검토) |
| 전자상거래 결제·공급·청약철회 기록 | **5년**(전자상거래법 시행령 제6조) |

**금지선**
- retention_class 없는 데이터 생성 / 파기 배치 없이 원시 이벤트 무기한 보관.

**준수증거**
- 파기 배치(cron) 존재·실측 / retention_class 컬럼 not-null / purge 후 잔존 0 검증.

## 제6조 — 자동 판정은 flag만, 자동 제재 금지

**원칙**
- 악성 판단에 **원시 신고수·추천수를 점수로 쓰지 않는다**(집단·보복신고에 취약). **인용(認容) 신고·서로 다른 인증회원 신고자·재범 여부·이의제기 결과**로 판단.
- 자동 위험점수는 점수만 표시 말고 **산출 근거 표시**, **자동 제재 금지**, **인간 검토 필수**, 이의제기 가능, 모델·가중치 변경 이력 보존.
- 추천수는 콘텐츠 기여도 참고·스팸계정 판별 보조·참여 추세 정도로만(추천 많다고 제재 완화, 적다고 위험 취급 금지).

**금지선**
- 신고 수만으로 자동 제재 / 추천수로 제재 완화·강화 / 산출 근거 없는 위험점수.

**준수증거**
- 모든 제재 레코드에 사람 actor_id + 사유. 자동 제재 경로 0. 위험 플래그는 큐잉만.

## 제7조 — 정직한 익명 고지 + 동의

**원칙**
- 익명의 사용자 약속을 정직하게 정의·고지: "다른 이용자에게는 작성자의 계정·닉네임·실명이 보이지 않는다. 운영진도 평상시 모든 익명글 작성자를 자유롭게 열람할 수 없다. 신고·안전·법적 대응 등 정당한 사건이 발생하면 제한된 운영자가 감사기록을 남기고 절차에 따라 확인할 수 있다."
- 활동정보 수집·처리방침을 가입 시 고지·동의. "완전 익명"이라고 오도하지 않는다.

**금지선**
- "완전 익명" 오도 문구 / 처리방침 없는 활동데이터 수집 / 고지 없는 목적 확대.

**준수증거**
- 처리방침·이용약관 문서 존재. 익명 UI 라벨이 위 문구와 일치. 동의 기록.

## 제8조 — 권한표 + 접근 audit (owner ⊇ operator)

**원칙**
- 역할 판정은 단일 소스(`lib/policy/permissions.js` + DB 정책 동일 기준). owner ⊇ operator ⊇ moderator ⊇ member ⊇ guest.
- 민감 조회(모더레이션 카드·신원공개·결제·분석 개별조회)는 **접근 자체를 audit**한다.

**금지선**
- 권한표 밖 상시 열람 화면 / audit 없는 민감 조회 / service_role로 권한 우회.

**준수증거**
- 권한 격리 테스트(anon·authenticated·moderator·operator 거부/허용) / 민감 조회 audit 레코드 / 우회 RPC 0.

## 부록 — SNUE 소규모 재식별 주의
단일 교대(과×학년×학기 코호트 극소)라 가명·집계여도 특정 가능. 분석 코호트 소규모 셀 억제(k 임계), Tier1 카드 희귀속성 조합 회피, 안정 크로스세션 actor id는 꼭 필요할 때만·가능하면 세션스코프.
