# 소셜 로그인 붙이기 — 키 발급 체크리스트

> 코드는 제가 만들 수 있지만 **키 발급은 상호님 손이 필요합니다**(카카오·구글이
> 계정 소유자 본인 확인을 요구). 이 문서는 그 10~20분짜리 작업의 순서표입니다.
>
> ⚠️ **Gate 4a 운영배포 전에는 붙이지 마세요.** 감사보고서 R2:
> 회원모델 없이 카카오를 붙이면 **한 사람이 이메일·카카오·구글로 각각 별개 회원**이
> 되어 데이터에 고착되고, 나중에 합치는 건 계정 탈취 위험 때문에 사실상 불가능합니다.
> 중복가입의 최종 방어선은 이메일이 아니라 **학번 HMAC**이라, 그 구조가 먼저 있어야 합니다.

## 0. 하기 전에

- [ ] Gate 4a 운영배포 완료 (회원·역할·신원 모델이 운영에 있어야 함)
- [ ] 지금 로그인 방식(이메일 매직링크)이 정상 동작하는지 확인

## 1. 카카오 (약 10분)

1. [ ] [카카오 개발자센터](https://developers.kakao.com) 로그인 → **내 애플리케이션** → 애플리케이션 추가
   - 앱 이름: `서울교대 새록이` / 사업자명: 개인 이름으로 가능
2. [ ] **앱 키** 화면에서 `REST API 키` 복사 → 안전한 곳에 (채팅에 붙여넣지 마세요)
3. [ ] **카카오 로그인** 메뉴 → 활성화 ON
4. [ ] **Redirect URI** 등록:
   ```
   https://jclwkvxbvsegmbcnptpi.supabase.co/auth/v1/callback
   ```
5. [ ] **동의항목** → 닉네임(필수 아님), 이메일(선택)
   - ⚠️ 이메일 동의를 안 하는 사용자가 있습니다. 우리 모델은 이메일을 회원 식별에
     쓰지 않으므로 차단 사유는 아니지만, 복구 수단 확보를 위해 가입 후
     "이메일 연결" 배너를 띄웁니다.
6. [ ] **보안** → Client Secret 생성 → 복사
7. [ ] Supabase 대시보드 → Authentication → Providers → **Kakao** → 활성화
   - REST API 키 → `Client ID`
   - Client Secret → `Client Secret`

## 2. 구글 (약 10분)

1. [ ] [Google Cloud Console](https://console.cloud.google.com) → 새 프로젝트
2. [ ] **API 및 서비스 → OAuth 동의 화면** → 외부 → 앱 이름·지원 이메일 입력
3. [ ] **사용자 인증 정보 → 사용자 인증 정보 만들기 → OAuth 클라이언트 ID**
   - 유형: 웹 애플리케이션
   - 승인된 리디렉션 URI:
     ```
     https://jclwkvxbvsegmbcnptpi.supabase.co/auth/v1/callback
     ```
4. [ ] 클라이언트 ID·보안 비밀번호 복사
5. [ ] Supabase → Authentication → Providers → **Google** → 활성화 후 붙여넣기

## ⚠️ 동일 이메일 자동 연결 (반드시 알아둘 것)

Supabase는 **검증된 동일 이메일을 자동으로 한 계정에 연결**합니다. 이걸 끄는 설정은
공식 문서에 없습니다. 다만 우리 설계의 불변식은 그대로입니다:

> **이메일·로그인수단은 학생 신원의 증거가 아니다.**
> verified는 오직 증빙 제출 → 운영자 승인 → 학번 HMAC 중복검사 통과로만 부여된다.

자동 연결의 실제 동작(매직링크↔구글↔카카오 교차, 이메일 미제공·미검증, 타 계정에
기연결된 identity 재연결)은 **dev 프로젝트에서 전수 테스트한 뒤에** 단정해야 합니다.
"동일 이메일이면 반드시 한 계정"이라고 가정한 코드를 쓰지 마세요.

## 키 취급

- REST API 키·Client Secret은 **Supabase 대시보드에만** 입력합니다
- `.env`·저장소·채팅·스크린샷에 넣지 않습니다
- 실수로 노출되면 즉시 재발급 (양쪽 콘솔에서 회전 가능)
